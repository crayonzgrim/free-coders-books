# 현업에서 통하는 코딩 습관

코딩을 잘하는 것과 **현업에서 일을 잘하는 것**은 다릅니다. 학생 때는 혼자 문제를 푸는 것이 중요하지만, 회사에서는 **팀과 함께 가치를 만드는 것**이 중요합니다.

## 읽기 좋은 코드 작성하기

### 코드는 쓰는 것보다 읽히는 횟수가 많다

당신이 작성한 코드를 읽을 사람:
- 코드 리뷰어
- 미래의 당신
- 새로 합류한 팀원
- 버그 수정하는 사람

그래서 **읽기 쉬운 코드**가 중요합니다.

### 자기 설명적 코드

```javascript
// 나쁜 예
const d = new Date();
const x = d.getTime() - s.getTime();
if (x > 86400000) { ... }

// 좋은 예
const currentDate = new Date();
const timeDifferenceMs = currentDate.getTime() - startDate.getTime();
const ONE_DAY_MS = 24 * 60 * 60 * 1000;

if (timeDifferenceMs > ONE_DAY_MS) { ... }
```

### 함수와 변수명에 신경 쓰기

```javascript
// 나쁜 예
function calc(a, b) { ... }
function doStuff(data) { ... }

// 좋은 예
function calculateTotalPrice(items, taxRate) { ... }
function formatUserDisplayName(user) { ... }
```

## 커밋 습관

### 작은 단위로 자주

하루 종일 작업하고 한 번에 커밋하지 마세요.

```
# 나쁜 예
git commit -m "여러 기능 구현"

# 좋은 예
git commit -m "feat: 로그인 폼 UI 구현"
git commit -m "feat: 로그인 API 연동"
git commit -m "fix: 비밀번호 유효성 검사 오류 수정"
git commit -m "test: 로그인 컴포넌트 테스트 추가"
```

### 커밋 메시지는 명확하게

미래의 누군가(또는 당신)가 "왜 이 변경을 했지?"를 물을 때 답이 되어야 합니다.

```
# 나쁜 예
"fix bug"
"update"
"..."

# 좋은 예
"fix: 모바일에서 드롭다운 메뉴가 닫히지 않는 버그 수정"
"refactor: 중복된 유효성 검사 로직을 utils로 분리"
"docs: API 응답 형식 문서 업데이트"
```

## 코드 리뷰 주고받기

### 리뷰 요청할 때

**1. 셀프 리뷰 먼저**

PR 올리기 전에 자신의 코드를 먼저 검토하세요. 뻔한 실수는 미리 잡읍시다.

**2. 맥락 설명**

"뭘 왜 이렇게 했는지" 설명해주세요.

```markdown
## 변경 내용
장바구니 담기 버튼 클릭 시 로딩 상태를 표시합니다.

## 변경 이유
- 느린 네트워크에서 사용자가 버튼이 작동하는지 알 수 없었음
- 연속 클릭으로 중복 요청이 발생하는 문제도 함께 해결

## 고민한 점
로딩 스피너 vs 버튼 비활성화 중 비활성화를 선택했습니다.
(스피너는 기존 디자인과 맞지 않아서)
```

**3. 작은 단위로**

500줄짜리 PR은 리뷰하기 싫습니다. 가능하면 100줄 내외로 나누세요.

### 리뷰할 때

**1. 비판이 아닌 제안**

```
# 나쁜 예
"이건 왜 이렇게 했어요?"
"이거 별로인데..."

# 좋은 예
"이 부분을 A 방식으로 하면 B 장점이 있을 것 같은데, 어떻게 생각하세요?"
"여기서 C 케이스는 어떻게 처리되나요?"
```

**2. 칭찬도 하기**

좋은 코드는 칭찬해주세요. 동기부여가 됩니다.

```
"이 추상화 좋네요!"
"깔끔하게 잘 정리되어 있네요 👍"
```

## 문서화 습관

### README 필수

```markdown
# 프로젝트 이름

## 소개
이 프로젝트가 뭘 하는지 한 줄로

## 설치 방법
npm install

## 실행 방법
npm run dev

## 환경 변수
필요한 환경 변수 목록

## 기여 방법
PR 환영합니다!
```

### 코드 주석은 "왜"를 설명

```javascript
// 나쁜 예 - "무엇"을 설명
// 가격을 10% 할인한다
const discountedPrice = price * 0.9;

// 좋은 예 - "왜"를 설명
// 2026년 2월 프로모션: 신규 회원 10% 할인 (PROMO-2026-02)
const discountedPrice = price * 0.9;
```

## 커뮤니케이션

### 막히면 빨리 물어보기

혼자 3시간 삽질하지 마세요. 15분 고민하고 안 되면 물어보세요.

단, **잘 물어보기**가 중요합니다:

```
# 나쁜 예
"이거 안 돼요 ㅠㅠ"
"에러 나요"

# 좋은 예
"A 기능을 구현하는데, B를 시도했더니 C 에러가 발생합니다.
구글링해서 D를 해봤는데 같은 에러예요.
혹시 비슷한 경험 있으신가요?"
```

### 진행 상황 공유

특히 리모트 환경에서 중요합니다.

```
오전: "오늘 A 기능 작업 시작합니다"
점심: "A 기능 70% 완료, 예상보다 B 부분이 복잡해서 좀 더 걸릴 것 같습니다"
퇴근: "A 기능 완료, PR 올렸습니다. 내일 테스트 예정"
```

## 마무리

좋은 개발자는 코딩만 잘하는 사람이 아닙니다:

- **팀과 소통**하고
- **다른 사람이 읽기 좋은 코드**를 작성하며
- **문서화**를 게을리하지 않고
- **겸손하게 배우는** 사람입니다

이런 습관은 하루아침에 만들어지지 않습니다. 오늘부터 하나씩 실천해보세요!
